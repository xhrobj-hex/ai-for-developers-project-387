# Hexlet brief — календарь звонков / booking slots

## Что это за проект

Это учебный проект курса Hexlet «AI for Developers»: нужно сделать **рабочую систему бронирования временных слотов** по мотивам **Cal.com**.

Ключевая идея проекта — не вручную писать каждую часть системы, а **довести продукт до рабочего состояния через агентный подход**. Важен не «идеальный» код сам по себе, а умение управлять разработкой с помощью ИИ, сохраняя фокус на результате. Основа проекта — подход **Design First**: сначала контракт, потом реализация. 

## Что должно получиться

Нужно пройти путь:

**TypeSpec → OpenAPI → backend + frontend → e2e → CI → Docker → deploy**.

На практике это означает, что у проекта должны появиться:

- контракт API, описанный через TypeSpec;
- OpenAPI-спека, сгенерированная из TypeSpec;
- frontend для основного пользовательского сценария;
- backend, реализующий этот контракт;
- e2e-проверки главного потока;
- базовый CI;
- контейнерный запуск и деплой.

## Рабочий принцип

Проект выполняется по принципу **Design First**:

1. сначала фиксируется MVP и контракт;
2. затем строятся backend и frontend;
3. потом подтверждается сквозная работоспособность;
4. после этого добавляются e2e, CI, Docker и deploy.

Это важно, потому что здесь контракт — источник истины для всей системы. Реализация должна подстраиваться под него, а не наоборот.

## Зафиксированный MVP проекта

Текущий MVP уже определён:

- есть один заранее заданный владелец календаря;
- владелец управляет типами событий;
- гость без регистрации открывает публичную страницу записи;
- гость видит доступные слоты на ближайшие 14 дней;
- гость бронирует свободный слот;
- система запрещает двойное бронирование одного и того же времени;
- владелец видит список предстоящих встреч.

## Что не входит в первую версию

В MVP не входят:

- регистрация и авторизация;
- несколько владельцев;
- интеграции с внешними календарями;
- уведомления и напоминания;
- оплаты;
- видеоссылки;
- сложное управление доступностью;
- продвинутый UI и второстепенные функции.

## Критерий готовности первой версии

Первая версия считается готовой, если гость может через интерфейс выбрать тип события и забронировать свободный слот в пределах 14 дней, система блокирует повторное бронирование занятого времени, а владелец видит созданную запись в списке предстоящих встреч.

## Как использовать этот документ

Этот файл нужен как короткая инициализирующая справка для агента:

- что это за проект;
- зачем он делается;
- какой у него учебный смысл;
- какой результат считается достаточным;
- где заканчивается MVP.

Если возникают сомнения по новой идее, сначала нужно проверить её на вопрос:

**это нужно для первой рабочей версии или нет?**
