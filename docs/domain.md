# Domain notes — календарь звонков / booking slots

## Назначение документа

Этот файл фиксирует предметную область MVP простым инженерным языком: какие сущности есть в системе, как они связаны и какой главный сценарий должен поддерживаться.

Документ не заменяет контракт API, а помогает согласовать смысловые договорённости перед реализацией.

## Суть продукта

Система позволяет гостю выбрать тип события, увидеть свободные временные слоты и забронировать подходящее время у владельца календаря.

В первой версии система упрощённая:

- есть только **один владелец календаря**;
- гость действует **без регистрации**;
- система работает в рамках **окна доступности на 14 дней**;
- главное действие системы — **создать бронь на свободный слот и не допустить повторное бронирование того же времени**.

## Роли

### Владелец календаря

Владелец:

- создаёт и редактирует типы событий;
- просматривает список предстоящих бронирований;
- является получателем всех созданных записей.

В MVP владелец заранее задан и не требует отдельного сценария регистрации или логина.

### Гость

Гость:

- открывает публичную страницу записи;
- выбирает тип события;
- просматривает свободные слоты;
- бронирует подходящее время.

В MVP у гостя нет аккаунта и личного кабинета.

## Основные сущности

### 1. EventType

Тип события, который создаёт владелец.

Минимальные поля для MVP:

- `id`
- `name`
- `description`
- `duration`

Смысл:

- определяет, какую встречу можно забронировать;
- определяет длительность будущей записи;
- отображается гостю на публичной странице.

### 2. Slot

Временной слот, доступный для бронирования.

Смысл:

- это потенциальный интервал времени, который система может показать гостю как свободный;
- слот существует в пределах ближайших 14 дней;
- слот связан с конкретным типом события через длительность;
- слот может быть **свободным** или **занятым**.

Важно: в реализации слот может быть как заранее сохранённой сущностью, так и вычисляемым представлением доступного времени. Для предметной области важно только то, что пользователь видит список доступных интервалов и выбирает один из них.

### 3. Booking

Бронирование, созданное гостем.

Минимальный смысл для MVP:

- относится к одному `EventType`;
- занимает один выбранный интервал времени;
- после создания делает этот интервал недоступным для повторной записи;
- отображается владельцу в списке предстоящих встреч.

## Связи между сущностями

Связи в MVP такие:

- один владелец имеет много `EventType`;
- один `EventType` может иметь много потенциальных `Slot`;
- один `Slot` может быть использован максимум в одном `Booking`;
- один `Booking` всегда создаётся для одного `EventType` и одного выбранного временного интервала.

## Главный пользовательский сценарий

1. Владелец создаёт один или несколько типов событий.
2. Гость открывает публичную страницу записи.
3. Гость выбирает нужный тип события.
4. Система показывает свободные слоты на ближайшие 14 дней.
5. Гость выбирает слот.
6. Система создаёт бронирование.
7. Система блокирует повторное использование этого времени.
8. Владелец видит новую запись в списке предстоящих встреч.

## Ключевые бизнес-правила

### Окно записи

Пользователь может бронировать только в пределах **14 дней от текущей даты**. Слоты вне этого окна в MVP не показываются.

### Отсутствие двойного бронирования

Два бронирования не могут занять одно и то же время. Если слот уже занят, система должна отклонить повторную попытку записи.

### Один владелец

В системе нет мульти-tenant-сценария. В MVP не нужно моделировать нескольких владельцев календаря.

### Гость без аккаунта

Создание бронирования не должно зависеть от регистрации, логина или личного кабинета.

## Что важно для контракта API

На уровне контракта должны быть покрыты как минимум такие возможности:

- получить список типов событий;
- создать тип события;
- обновить тип события;
- получить список доступных слотов для выбранного типа события;
- создать бронирование на свободный слот;
- получить список предстоящих бронирований владельца.

Названия конкретных endpoint'ов и shape DTO определяются в TypeSpec.

## Что сознательно не моделируем в MVP

Пока не нужно тащить в предметную область:

- пользователей и роли со сложной авторизацией;
- внешние календари;
- нотификации;
- оплату;
- видеоконференции;
- сложные правила доступности;
- повторяющиеся события и расширенные расписания;
- временные зоны как отдельную большую подзадачу.

## Практический смысл для Codex

Если при реализации появляются спорные решения, приоритет такой:

1. не ломать зафиксированный MVP;
2. не усложнять модель без необходимости;
3. сначала обеспечить главный сценарий;
4. только потом думать о расширениях.

Этот документ должен помогать агенту удерживать фокус на предметной области, а не расползаться в необязательные фичи.
