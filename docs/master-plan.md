# Master plan проекта: календарь звонков / booking slots

## Назначение документа

Этот файл — верхнеуровневый план проекта.  

Он нужен для общего управления работой: понимать текущий этап, следующую веху, критерии завершения и не расползаться в детали раньше времени.

---

## 0. Общая цель проекта

Сделать **рабочую систему бронирования временных слотов** по принципу **Design First**:

**TypeSpec → OpenAPI → backend + frontend → e2e → CI → Docker → deploy**

Главная цель проекта — не вручную отполировать код, а **довести систему до рабочего состояния через агентный подход**.

---

## 1. Глобальные этапы проекта

### Этап 0. Зафиксировать цель и границы MVP

**Цель:** понять, что именно делаем в первой рабочей версии.

**Что входит:**

- краткое описание продукта;
- основной пользовательский сценарий;
- список MVP-функций;
- список того, что пока не входит.

**Результат этапа:**

- есть формулировка MVP;
- есть критерий «первая версия готова».

**Не делать на этом этапе:**

- не уходить в реализацию;
- не обсуждать глубоко UI;
- не спорить о второстепенных фичах.

**Критерий завершения:**

- можно одним абзацем объяснить, что умеет первая версия системы.

---

### Этап 1. Спроектировать контракт системы

**Цель:** описать систему через контракт, а не через код.

**Что входит:**

- определить сущности;
- определить действия пользователя и системы;
- описать API в TypeSpec;
- сгенерировать OpenAPI.

**Результат этапа:**

- есть TypeSpec;
- есть OpenAPI-спека;
- понятны входы, выходы и основные сценарии.

**Не делать на этом этапе:**

- не писать полноценный backend;
- не делать UI;
- не уходить в инфраструктуру глубже минимума.

**Критерий завершения:**

- по спецификации можно начинать реализацию backend и frontend.

---

### Этап 2. Собрать технический каркас

**Цель:** сделать проект организованным и запускаемым.

**Что входит:**

- выбрать стек frontend;
- выбрать стек backend;
- выбрать БД;
- определить структуру репозитория;
- подготовить заготовки приложений;
- настроить локальный запуск;
- подготовить env-конфиги;
- оформить README.

**Результат этапа:**

- проект стартует локально;
- структура каталогов понятна;
- агенту и человеку удобно продолжать разработку.

**Не делать на этом этапе:**

- не реализовывать всю бизнес-логику;
- не пытаться закрыть весь продукт сразу.

**Критерий завершения:**

- новый человек может понять структуру проекта и поднять его локально.

---

### Этап 3. Реализовать backend MVP

**Цель:** поднять серверную часть для главного сценария.

**Что входит:**

- минимальная доменная модель;
- хранение данных;
- API по контракту;
- базовая валидация;
- минимальная обработка ошибок.

**Результат этапа:**

- backend обслуживает основной сценарий;
- API соответствует контракту.

**Не делать на этом этапе:**

- не добавлять второстепенные функции;
- не пытаться сделать «идеальную архитектуру».

**Критерий завершения:**

- backend можно отдельно проверить запросами и получить ожидаемый результат.

---

### Этап 4. Реализовать frontend MVP

**Цель:** собрать минимальный пользовательский интерфейс для основного сценария.

**Что входит:**

- экран(ы) для главного сценария;
- подключение к API;
- базовая обработка состояний;
- минимально приемлемый UX.

**Результат этапа:**

- пользователь может пройти основной сценарий через браузер.

**Не делать на этом этапе:**

- не полировать UI раньше времени;
- не добавлять украшения вместо функциональности.

**Критерий завершения:**

- основной сценарий можно пройти вручную от начала до конца.

---

### Этап 5. Сквозная интеграция

**Цель:** убедиться, что frontend, backend и БД работают как единая система.

**Что входит:**

- интеграционная проверка всего потока;
- исправление разрывов между слоями;
- выравнивание контрактов, конфигов и данных.

**Результат этапа:**

- система работает локально целиком.

**Критерий завершения:**

- основной сценарий проходит вручную без костылей.

---

### Этап 6. Написать e2e-тесты

**Цель:** автоматически подтвердить, что система реально работает.

**Что входит:**

- e2e-тест на главный сценарий;
- 1–2 негативных сценария;
- воспроизводимый запуск тестов.

**Результат этапа:**

- есть автоматическая проверка MVP.

**Критерий завершения:**
- e2e-тесты стабильно проходят локально.

---

### Этап 7. Подключить CI

**Цель:** автоматизировать проверки.

**Что входит:**

- запуск линтеров / базовых проверок;
- запуск тестов;
- при необходимости — сборка проекта;
- фиксация минимального quality gate.

**Результат этапа:**

- проект проверяется автоматически на каждом изменении.

**Критерий завершения:**

- после push / PR проверки запускаются без ручного участия.

---

### Этап 8. Упаковать проект в Docker

**Цель:** сделать систему воспроизводимой и удобной для доставки.

**Что входит:**

- Dockerfile / Dockerfiles;
- docker compose или другой способ локального оркестрования;
- проверка контейнерного запуска.

**Результат этапа:**

- проект можно уверенно поднимать в контейнерах.

**Критерий завершения:**

- локальный запуск в контейнерах воспроизводим.

---

### Этап 9. Выполнить deploy

**Цель:** опубликовать рабочую версию проекта.

**Что входит:**

- выбор площадки;
- выкладка frontend/backend;
- настройка переменных окружения;
- проверка рабочей ссылки.

**Результат этапа:**

- проект доступен по URL.

**Критерий завершения:**

- основной сценарий проходит уже в задеплоенной версии.

---

### Этап 10. Стабилизация и улучшения

**Цель:** сделать проект более надёжным и презентабельным.

**Что входит:**

- фиксы багов;
- улучшение UX/UI;
- второстепенные фичи;
- улучшение документации.

**Результат этапа:**

- проект выглядит завершённым, а не просто «как-то работает».

**Критерий завершения:**

- критичных дефектов нет, демо показывать не стыдно.

---

## 2. Крупные вехи

### Веха A. Проект определён

Есть:

- понятная цель;
- границы MVP;
- контракт.

### Веха B. Проект ожил локально

Есть:

- каркас;
- backend MVP;
- frontend MVP;
- ручной проход главного сценария.

### Веха C. Проект подтверждён автоматически

Есть:

- e2e;
- CI;
- контейнерный запуск.

### Веха D. Проект опубликован

Есть:

- deploy;
- рабочая ссылка;
- стабилизированная версия.
