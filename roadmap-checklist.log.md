# Проходждение roadmap (cal-booking-roadmap.md)

## Этап 0. Зафиксировать цель и границы MVP

### MVP проекта

Календарь звонков — это упрощённый сервис бронирования времени по мотивам Cal.com. В системе есть две роли: владелец календаря и гость. Владелец управляет типами событий и просматривает предстоящие записи. Гость без регистрации выбирает тип события, видит свободные слоты на ближайшие 14 дней и бронирует подходящее время.

### Основной сценарий

1. Владелец создаёт один или несколько типов событий.
2. Гость открывает публичную страницу записи.
3. Гость выбирает тип события.
4. Система показывает свободные слоты на ближайшие 14 дней.
5. Гость выбирает слот и создаёт бронь.
6. Система подтверждает запись и не позволяет занять это же время повторно.
7. Владелец видит новую запись в списке предстоящих встреч.

### Что входит в MVP

- один заранее заданный владелец календаря;
- создание типов событий (`id`, `name`, `description`, `duration`);
- публичная страница со списком типов событий;
- просмотр свободных слотов для выбранного типа события;
- окно записи на 14 дней от текущей даты;
- создание бронирования на свободный слот;
- запрет на две записи в одно и то же время;
- просмотр владельцем списка предстоящих встреч.

### Что не входит в MVP

- регистрация и авторизация;
- личные кабинеты;
- несколько владельцев;
- интеграции с внешними календарями;
- уведомления и напоминания;
- оплаты и видеоссылки;
- сложный редактор доступности;
- продвинутый UI и второстепенные фичи.

### Критерий готовности первой версии

Первая версия готова, если гость может через интерфейс выбрать тип события и успешно забронировать свободный слот в пределах 14 дней, система блокирует повторное бронирование занятого времени, а владелец видит созданную запись в списке предстоящих встреч.

## Этап 1. Спроектировать контракт системы

### Что сделали

- Зафиксировали контракт системы по принципу Design First.
- Сверили контракт с утверждённым MVP и не стали расширять его сверх согласованных границ.
- Подготовили TypeSpec-спецификацию API в файле `spec/main.tsp`.
- Сгенерировали OpenAPI-спеку в файле `spec/openapi.yaml`.
- Добавили минимальную инфраструктуру для работы с TypeSpec/OpenAPI:
  - `package.json`
  - `package-lock.json`
  - `tspconfig.yaml`
- Обновили `.gitignore`, чтобы не тащить в репозиторий `node_modules/`.

### Что зафиксировано в контракте

#### Сущности

- тип события;
- слот;
- бронирование;
- владелец календаря как заранее заданный актор системы.

#### Основные сценарии

- владелец создаёт типы событий;
- владелец просматривает список предстоящих встреч;
- гость видит список типов событий;
- гость получает свободные слоты на ближайшие 14 дней;
- гость создаёт бронирование на выбранный слот.

#### Ограничения

- в MVP нет регистрации и авторизации;
- владелец один и заранее задан;
- окно записи ограничено ближайшими 14 днями;
- нельзя создать две записи на одно и то же время, даже если это разные типы событий.

### Что не делали на этом этапе

- не расширяли MVP второстепенными фичами.

## Этап 3. Реализовать backend MVP

### Что сделали

- Реализовали backend MVP строго по уже зафиксированному контракту из TypeSpec/OpenAPI.
- Добавили in-memory хранение для типов событий и бронирований без отдельной БД.
- Подключили контрактные backend endpoint'ы:
  - `GET /event-types`
  - `GET /event-types/{eventTypeId}/slots`
  - `POST /bookings`
  - `POST /admin/event-types`
  - `GET /admin/bookings/upcoming`
- Сохранили технический endpoint `GET /health` для локальной проверки сервера.
- Добавили минимальные доменные модели и ошибки backend-слоя.
- Реализовали базовую валидацию запросов и маппинг ошибок в контрактные HTTP-ответы.
- Реализовали запрет двойного бронирования одного и того же временного интервала.
- Добавили генерацию свободных слотов на ближайшие 14 дней по зафиксированным MVP-допущениям.
- Прогнали `gofmt` и `go test ./...`.
- Выполнили ручную HTTP-проверку основного сценария MVP.
- Добавили минимальный HTTP-level тест на сценарий конфликта повторного бронирования.

### Какие MVP-допущения зафиксировали на этом этапе

Так как контракт не задавал полные правила генерации доступности, для backend MVP приняли минимальные рабочие допущения:

- свободные слоты генерируются только на ближайшие 14 дней;
- рабочее окно: каждый день с `09:00` до `18:00`;
- шаг сетки слотов: `30` минут;
- бронь можно создать только на слот из этой сетки;
- `endAt` вычисляется как `startAt + durationMinutes`;
- конфликт определяется пересечением временных интервалов;
- данные хранятся только in-memory и сбрасываются после рестарта процесса.

### Что получилось на выходе

- backend поднимается локально и обслуживает основной сценарий MVP;
- типы событий можно создавать и получать списком;
- для типа события можно получить свободные слоты;
- на свободный слот можно создать бронь;
- повторная бронь на то же время отклоняется с `409 SLOT_ALREADY_BOOKED`;
- владелец может получить список предстоящих встреч;
- базовые негативные сценарии покрыты ответами `400`, `404`, `409`, `422`.

## Этап 4. Реализовать frontend MVP

### Что сделали

- Реализовали frontend MVP на React + TypeScript + Vite поверх уже готового backend API.
- Подключили frontend к backend без расширения контракта сверх согласованных границ MVP.
- Настроили локальный dev proxy через Vite, чтобы frontend в dev-режиме обращался к backend через same-origin путь `/api`.
- Реализовали публичный flow:
  - главная страница `/` загружает список типов событий через `GET /event-types`;
  - страница `/book/:eventTypeId` загружает свободные слоты через `GET /event-types/{eventTypeId}/slots`;
  - страница `/book/:eventTypeId/confirm` подтверждает выбранный слот и создаёт бронь через `POST /bookings`.
- Реализовали owner-часть на `/admin`:
  - создание типа события через `POST /admin/event-types`;
  - просмотр предстоящих записей через `GET /admin/bookings/upcoming`.
- Добавили базовые пользовательские состояния для экранов:
  - `loading`, `empty`, `error`, `success` там, где это требуется по сценарию;
  - `submitting`, `success`, `error` для подтверждения бронирования;
  - отдельную обработку `409 SLOT_ALREADY_BOOKED` и `422 BOOKING_RULE_VIOLATION` на confirm screen.
- Добавили controlled error state при прямом входе на экран подтверждения без выбранного слота.
- Явно показали во frontend текущие operational rules MVP:
  - ближайшие 14 дней;
  - окно `09:00–18:00 UTC`;
  - шаг сетки `30` минут.
- Обновили README и статус проекта под фактическое завершение frontend MVP.

## Этап 5. Сквозная интеграция frontend и backend

### Что сделали

- Проверили сквозную интеграцию frontend и backend на реальном локальном запуске.
- Подтвердили, что на текущем этапе роль storage выполняет in-memory хранилище внутри backend-процесса, без отдельной внешней БД.
- Подняли backend локально на `:8080`.
- Подняли frontend через Vite dev server на `127.0.0.1:5173`.
- Проверили, что dev proxy ` /api -> http://localhost:8080 ` работает корректно для чтения и записи.
- Прогнали основной сценарий MVP сквозным образом через frontend-origin путь `/api`:
  - создание типа события;
  - получение списка типов событий;
  - получение свободных слотов;
  - создание бронирования;
  - повторная попытка бронирования того же времени;
  - получение списка upcoming bookings.
- Дополнительно подтвердили важный интеграционный инвариант: после успешного booking занятый слот исчезает из списка свободных слотов.
- Выполнили ручную smoke-проверку в браузере: owner flow на `/admin` и guest flow на публичной части проходят без явных ошибок.

### Что получилось на выходе

- frontend и backend реально работают как единая система;
- owner может создать тип события через `/admin`;
- guest видит созданный тип события на публичной странице;
- guest видит свободные слоты и может создать booking;
- повторная бронь того же времени блокируется ответом `409 SLOT_ALREADY_BOOKED`;
- после бронирования занятый слот исчезает из списка свободных;
- owner видит созданную запись в `upcoming bookings`;
- основной сценарий MVP проходит вручную без костылей.

### Что было сломано

Существенных интеграционных дефектов между frontend и backend не обнаружено.

Во время проверки возник один ложный сигнал: первый `GET /event-types` был выполнен параллельно с `POST /admin/event-types`, из-за чего создалось впечатление, что запись не сохранилась. Повторная последовательная проверка показала, что это была гонка в самой проверке, а не проблема кода.

Файловые изменения по итогам сквозной интеграции не понадобились: текущая реализация уже проходила сценарий на локальном стенде.

#### 2. Duplicate booking

Проверяется негативный сценарий повторного бронирования:

- создаётся event type для теста;
- два guest-контекста доходят до подтверждения одного и того же слота;
- первая бронь проходит успешно;
- вторая попытка получает конфликт;
- UI показывает состояние `SLOT_ALREADY_BOOKED`.

#### 3. Invalid owner form submit

Проверяется негативный сценарий невалидной отправки owner-формы:

- owner открывает `/admin`;
- отправляет форму создания event type без валидных данных;
- UI показывает error state;
- новый event type не создаётся.
